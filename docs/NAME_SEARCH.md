# Hum Name Search

Date: 2026-07-06

## Recommendation

Use **Hum** as the working language name.

Rationale:

- Short and memorable.
- One syllable.
- Lives inside the word "human".
- Points directly at human-readable systems programming.
- Suggests code that is smooth enough to hum along with the machine.
- Does not box the language into being only for agents.
- Works as a file extension: `.hum`.
- Works as a compiler command: `humc`.
- Sits well next to Rust, Zig, Swift, Mojo, Carbon, Vale, and Hylo.

## Search Summary

`Vow` is rejected. User search found an existing result for "Vow -- A Programming Language for Agentic Coding" at `vow-lang.com`, which overlaps too directly with the agentic-contract language niche.

Initial web search did not reveal an obvious active systems programming language named `Hum`, `HumLang`, `hum-lang`, or `humc`.

Known collisions and caveats:

- `hum` exists as a PyPI package for audio synthesis: https://pypi.org/project/hum/
- `Hum` appears as an acronym/name in research papers and unrelated products.
- `hum.dev` is an unrelated developer hiring site.
- The word is common English, so package names, domains, products, or trademarks may exist outside programming-language search results.
- This is not legal or trademark clearance.

## Practical Naming Plan

Use:

```text
Language: Hum
Repository: hum-lang
Compiler: humc
CLI: hum
Extension: .hum
Package namespace: hum-lang or humpkg where exact hum is unavailable
```

If later trademark/package checks make `Hum` painful, the nearest fallback names are:

```text
Hume
Human
Hearth
Verity
```

Current preference is `Hum` because it is the clearest match for the language model: readable enough for humans, explicit enough for compilers, and structured enough for agents.
