# Shipping a Credible Windows-First Toolchain in 2026

For a new language like Hum, the credible 2026 distribution pattern is not “just publish a binary.” It is a release system: reproducible-enough builds, signed artifacts, provenance that ties the release to a specific source tree and build workflow, one or more machine-readable SBOMs, Windows-native package-manager paths, and an offline-verifiable install kit for regulated buyers. The evidence is strongest for a practical baseline of signed releases plus verifiable provenance and per-release checksums; it is weaker on any claim that security teams already require full bit-for-bit reproducibility or a single SBOM format for compiler toolchains specifically. The highest-risk mistake for a *new* toolchain is to ship only a GitHub release ZIP plus a checksum and call that “secure enough.” After xz and repeated package-ecosystem attacks, that is below the bar for many serious adopters. cite

## Reproducible release builds

**(a) Research signals.** **Finding — reproducible Rust binaries are attainable, but not turnkey. Confidence: high.** The Reproducible Builds project’s Rust guidance says Rust programs are “often already reproducible by default” if the original toolchain version is matched and build paths are normalized, but it immediately recommends `Cargo.lock`, `cargo build --locked`, path normalization, `SOURCE_DATE_EPOCH`, and `diffoscope` for real debugging because common variance still appears in practice. Rust’s own compiler documentation exposes `--remap-path-prefix` for normalizing source paths in emitted output. At the ecosystem level, openSUSE reported in February 2025 that its RBOS proof-of-concept reached 100% bit-identical packages for a usable distribution, showing that reproducibility is achievable at large scale with sustained patching and discipline. At the same time, active 2024–2026 Rust issues document remaining nondeterminism from codegen units and unstable build-folder content, which is strong evidence that “reproducible by default” is still too strong a promise for general Rust release pipelines. cite

**(b) Lessons for a new language.** **Finding — Hum should aim for “reproducible release policy,” not market day-one releases as universally reproducible. Confidence: high.** The right promise is narrower and testable: given a pinned toolchain, pinned dependencies, normalized paths, and a documented environment, Hum can rebuild the same release artifact or explain any remaining delta with tooling. That is much more credible than a broad claim of universal bit-for-bit reproducibility on all contributor machines. The openSUSE result shows that the end state is real; the Rust issues show that getting there requires systematic work, not slogans. **Thin evidence:** I did not find primary-source evidence that Windows-native Rust toolchain distributors broadly achieve zero-delta reproducibility today for every release artifact they ship. cite

**(c) Concrete design gates or tests to adopt.** **Finding — Hum should ship a reproducibility gate as part of release engineering. Confidence: high.** Concretely: release only from a pinned dependency graph; require a lockfile-equivalent; run two clean builds in separate directories and compare outputs with a diffoscope-style job; use path remapping; honor `SOURCE_DATE_EPOCH`; ban ambient timestamps in the compiler, stdlib packager, and installer metadata unless explicitly declared; and fail the release if the binary, embedded assets, or install bundle differ unexpectedly. For any non-zero diff, store the diff artifact and classify it. This is directly aligned with Rust/reproducible-builds guidance and with the kinds of failures still being found in Rust ecosystems. cite

**(d) What evidence would change the conclusion.** I would soften this conclusion if Hum’s implementation stack avoided the known Rust-like variance classes entirely and independent rebuilders could demonstrate sustained bit-for-bit agreement across Windows CI, Linux CI, and an external rebuilder set for several consecutive releases. I would harden it if Hum adopted self-hosted or workstation builds without provenance, or if early releases showed unexplained drift across nominally identical build environments.

## Signing provenance and transparency

**(a) Research signals.** **Finding — simple file signing alone is no longer enough for a new toolchain. Confidence: high.** Sigstore’s own documentation lays out the weakness of traditional long-lived verification-key distribution: identity ambiguity, key-management burden, and painful revocation. Its model instead binds short-lived certificates to identities and records signing events in Rekor, while the Sigstore bundle format packages the verification material, transparency-log data, and signature content needed for verification. GitHub’s artifact attestations now let projects generate provenance for binaries and signed SBOM attestations in GitHub Actions, verify them with `gh attestation verify`, and even verify them offline by carrying the artifact, bundle, and trusted roots into an air-gapped environment. SLSA’s current build-track guidance remains the best concise “what good looks like” rubric: L1 means provenance exists; L2 means the hosted build platform generates and signs provenance; L3 adds hardened isolation and keeps provenance-signing material inaccessible to build steps. cite

**(b) Lessons for a new language.** **Finding — the credible floor for Hum is a signed binary plus independently verifiable provenance; the likely target is SLSA Build L2, with L3 as a roadmap. Confidence: high.** A new language does not yet have brand trust, installed base, or widespread mirroring. That makes transparency more important, not less. **Inference, supported by SLSA and GitHub/Sigstore platform capabilities:** for a 2026 public alpha, Hum should treat “can verify exactly which workflow built this artifact” as the minimum bar for serious security buyers. Pure minisign-style signing is still useful for offline verification and simplicity, and Zig’s download page shows that it works in practice, but it does not by itself tell a consumer *which source, workflow, and builder identity* produced the artifact. That gap matters much more after xz. Also noteworthy: Zig’s own ecosystem had users explicitly asking for clearer minisign verification instructions, which is a quiet but important UX warning — if verification is not easy and obvious, most users will skip it. cite

**(c) Concrete design gates or tests to adopt.** **Finding — Hum should use dual trust signals. Confidence: high.** Ship every public release with two verifiers: a simple offline signature path and a provenance path. In practice that means (i) a conventional signed artifact path users can verify with minimal tooling, and (ii) a Sigstore/GitHub attestation bundle that lets security teams verify builder identity, repository, workflow, and subject digest. Make verification commands first-class in the docs, include expected certificate identity / OIDC issuer / repo / workflow, and test offline verification as a release gate. If Hum uses GitHub Actions, wire in `actions/attest@v4` for the binary and for SBOM attestation, and document the exact `gh attestation verify` command line consumers should run. For policy language, target SLSA Build L2 now and publish an explicit plan for L3-style isolation later. cite

**(d) What evidence would change the conclusion.** I would weaken this if major Windows security buyers started accepting unsigned-provenance-free toolchains again, or if a widely trusted Windows distribution channel gave small publishers first-class trust without separate provenance. I would strengthen it if more package ecosystems or enterprise controls began enforcing attestation verification by policy rather than leaving it optional.

## SBOM strategy for a compiler toolchain

**(a) Research signals.** **Finding — both SPDX and CycloneDX are credible in 2026, but they pull toward different buyer priorities. Confidence: high.** SPDX 3.0 was officially released in April 2024; its governance already sits on an ISO lineage, and the current spec family explicitly includes software build information. The SPDX 3.0.1 conformance text says the Build Profile can encode inputs, outputs, procedures, environments, and actors from the build process. CycloneDX, by contrast, is positioning itself as a full-stack BOM standard for cyber-risk reduction, with 1.7 released in October 2025 and the project highlighting vulnerability, VEX, services, and broader operational supply-chain use cases. GitHub’s attestation tooling explicitly supports either SPDX or CycloneDX SBOM predicates for verification. cite

**(b) Lessons for a new language.** **Finding — for a *compiler/toolchain*, the safest 2026 answer is to emit both formats from one canonical dependency graph. Confidence: medium-high.** SPDX has stronger compliance/procurement gravity because of its standardization history and build-profile direction; CycloneDX has stronger operational-security ergonomics because it is explicitly optimized for vulnerability, VEX, and broader risk workflows. **Thin evidence:** I did not find strong 2024–2026 primary-source evidence that buyers of compilers specifically prefer CycloneDX over SPDX or vice versa. The cleaner conclusion is that format duality is cheap insurance if your tooling can do it from one source of truth. If Hum must choose only one for an early alpha, the choice should follow target buyer: SPDX-first for procurement-heavy or legal/compliance-centric evaluation; CycloneDX-first for security-operations-heavy evaluation. cite

**(c) Concrete design gates or tests to adopt.** **Finding — attach the SBOM to the artifact, do not bury it in release notes. Confidence: high.** For Hum, emit an SBOM that covers at least the release binary, bundled libraries, installer/container metadata if any, the compiler’s dependency graph, and the build environment identity. Then sign or attest it against the exact shipped artifact digest. In GitHub terms, that means generating a JSON-formatted SBOM and an SBOM attestation bound to `subject-path`. Also test that the SBOM can be round-tripped or revalidated by at least two independent consumers. Because Hum is a toolchain, include build-tool and codegen inputs, not just runtime dependencies. cite

**(d) What evidence would change the conclusion.** If a target adoption segment — for example, defense primes or a specific enterprise procurement team — explicitly standardizes on one format, that should override the dual-format recommendation. Conversely, if Hum’s alpha footprint is very small and the engineering cost of maintaining two outputs is non-trivial, a single well-attested SBOM may be better than two low-quality ones.

## Windows-first distribution and reputation

**(a) Research signals.** **Finding — Windows distribution credibility is partly technical and partly reputational. Confidence: high.** On the technical side, WinGet manifests require `InstallerUrl` and `InstallerSha256`, and the ecosystem expects silent-install support. WinGet also supports secure trusted sources, explicit/private sources, and source export for Group Policy-style management. Scoop manifests are hash-driven and especially friendly to ZIP/portable distributions and shims. Chocolatey package tooling supports checksums, but Chocolatey’s own installation page also tells users to inspect remote install scripts before running them and provides explicit internal-repository guidance for organizations. On the distribution-model side, Deno’s installation docs still emphasize a single binary with no external dependencies and support installation via winget / Scoop / Chocolatey on Windows; Zig ships signed ZIPs and a machine-readable download page; Go still offers MSI and archive downloads with checksums. The lesson from these systems is consistent: publish a raw artifact path *and* integrate with native channels. cite

**(b) Lessons for a new language.** **Finding — Hum should ship four Windows surfaces at launch: raw ZIP/EXE, WinGet, Scoop, and Chocolatey. Confidence: high.** Raw artifacts are necessary for air-gapped and forensic workflows. WinGet is the lowest-friction Windows-native path for many developer and enterprise machines. Scoop is unusually good for portable, non-admin, single-binary and ZIP-centric tooling. Chocolatey remains important in enterprise automation, especially when organizations host packages internally. **Finding — do not force users through a PowerShell bootstrap script as the primary path. Confidence: high.** Chocolatey’s own install page effectively cautions users to inspect any internet-fetched script before execution, which is exactly how security reviewers think. A new language should make the script path secondary, not primary. cite

**(c) Concrete design gates or tests to adopt.** **Finding — Hum’s Windows package outputs should be treated as first-class release artifacts. Confidence: high.** Ship a signed ZIP containing the single `hum.exe` plus docs and verification material. Publish a WinGet manifest with `InstallerSha256`, silent install where relevant, and a package ID that will survive rebranding. Publish a Scoop manifest with hashes and portable shims. Publish a Chocolatey package that either embeds the installer or pins remote checksums, and separately support internalized packages for regulated customers. Test all four in CI on clean Windows 11 and Windows Server images, including non-admin portable install where applicable. cite

**(d) What evidence would change the conclusion.** If early adopters are overwhelmingly developers on managed Windows fleets that already standardize on one channel, Hum could concentrate there first. But absent that evidence, multi-channel Windows distribution is the safer adoption bet because each channel maps to a distinct trust workflow.

## SmartScreen and code-signing economics

**(a) Research signals.** **Finding — SmartScreen cold-start is now a file-reputation problem, not an EV-cert shortcut problem. Confidence: high.** Microsoft’s 2026 guidance is unusually direct: signed OV/EV apps can still show first-download warnings until reputation accumulates, and EV certificates no longer bypass SmartScreen. Microsoft explicitly says paying the EV premium solely to avoid SmartScreen warnings is no longer justified. The same docs recommend Microsoft’s Artifact Signing service for non-Store distribution, with much lower monthly starting cost than traditional certs, no hardware token, and pipeline-friendly integration — but regional availability remains limited, especially for individual developers. For traditional OV certificates, Microsoft’s guidance gives a typical range of about $150–300 per year and notes the HSM or hardware-token requirement stemming from CA/B Forum rules. Official Azure pricing shows Artifact Signing on a consumption model that starts much lower than typical public code-signing cert programs. Market pricing from major CAs is still materially higher for many plans, especially once hardware or annual-plan mechanics are included. cite

**(b) Lessons for a new language.** **Finding — for Hum, code signing is necessary but insufficient; SmartScreen warnings must be designed around, not wished away. Confidence: high.** That means Hum should expect warning friction on early downloads, even when correctly signed. The mitigation is not “buy EV and forget it”; Microsoft says that era is over. The mitigations are: sign everything; keep publisher identity stable; make the download source obviously official; accumulate clean installs; use package managers and trusted channels; and prefer Microsoft’s own signing path where region and buyer constraints allow. **Finding — solo-dev economics remain awkward. Confidence: medium-high.** For an incorporated organization in supported regions, Artifact Signing is cost-effective. For a solo developer outside Microsoft’s supported individual geographies, the current options look worse: OV certs need organization validation and hardware/HSM handling, while Microsoft’s individual path is region-limited. That is a real go-to-market constraint for a new Windows-first language. cite

**(c) Concrete design gates or tests to adopt.** **Finding — Hum should publish its Windows trust story explicitly. Confidence: high.** Release notes and docs should state the exact publisher identity users will see, how to verify the certificate, what a first-download SmartScreen warning may look like, and which official package IDs are valid in WinGet/Scoop/Chocolatey. Operationally, sign every EXE/MSI/ZIP-adjacent executable payload; do not mutate artifacts after signing; keep the publisher name stable across releases; and track SmartScreen incidence as a release metric. If Hum can use Artifact Signing, test it; if not, budget for OV signing and HSM handling. cite

**(d) What evidence would change the conclusion.** I would weaken the concern if Microsoft materially broadened individual Artifact Signing availability or introduced stronger trust transfer for new well-signed developer tools. I would strengthen it if SmartScreen treatment for low-volume developer tools grew harsher, or if enterprise buyers increasingly required vendor-specific CA relationships beyond Microsoft’s service.

## Air-gapped installs controlled updates and post-incident buyer expectations

**(a) Research signals.** **Finding — regulated buyers now expect offline-verifiable installation paths, not just downloadable hashes. Confidence: high.** GitHub’s attestation docs specifically support offline verification: download the attestation bundle and trusted roots online, then verify inside the disconnected environment. The docs also warn that trusted-root data can go stale because Sigstore key material rotates a few times per year. Cargo’s own docs support offline dependency workflows through `cargo fetch`, `cargo vendor`, and `--offline` / `--frozen`, which are relevant when a customer wants to build or audit the toolchain from source. Chocolatey’s organizational docs focus on repository solutions and internal hosting, and WinGet can add trusted sources, including URL or UNC-based sources, which matters for internal mirrors. Deno’s update command is also instructive: it exposes explicit version/channel selection, dry runs, and checksum verification instead of relying on an invisible background updater. cite

**(b) Lessons for a new language.** **Finding — Hum should avoid auto-update-by-default for the compiler/toolchain. Confidence: high.** Security buyers want controlled movement, not surprise mutation of their compiler. The better model is explicit channels — stable, preview, maybe canary — with user- or admin-initiated updates, checksum verification, and an option to mirror update metadata internally. **Finding — the air-gapped deliverable should be a kit, not a binary. Confidence: high.** For Hum that kit should include the artifact, signature, provenance bundle, SBOM, checksums, roots/trust snapshot, licenses/notices, and package-manager metadata or mirrored packages for internal distribution. **Inference, strongly supported by the sources:** the more complete the offline verification kit, the easier Hum becomes to evaluate in defense and regulated environments. cite

**(c) Concrete design gates or tests to adopt.** **Finding — Hum should add three release gates tied directly to the xz and npm era. Confidence: high.** First, require a “source-to-artifact” gate: the release tarball/ZIP, source snapshot, and attestation must all point to the same commit and subject digest. This is essential because the xz backdoor involved malicious content in distributed tarballs that did not match the upstream Git source in the expected way. Second, require a “no long-lived publish secret” gate for registry or package publishing wherever the ecosystem allows it; npm’s trusted publishing model exists precisely to eliminate reusable write tokens and now automatically emits provenance. Third, require an “offline verify” gate for each release candidate: rebuild the customer verification path in a disconnected test environment and record the exact commands. cite

**(d) What evidence would change the conclusion.** I would relax the offline-kit and controlled-update recommendations only if Hum’s target customers were clearly consumer/hobby users with no regulated deployment needs. I would strengthen them if Hum targets defense, industrial, or federal buyers, or if more organizations begin enforcing provenance and offline verification at intake.

The xz and npm incidents changed one thing most of all: **new toolchains must now prove not just that the bits are signed, but where the bits came from and how they were produced.** In xz, the problem was not merely a bad signature story; it was that the distributed release tarballs themselves were the attack vector and diverged from what many people assumed was “the source.” In the npm ecosystem, the platform response has moved sharply toward OIDC-based trusted publishing, automatic provenance, and signature/provenance verification (`npm audit signatures`) to reduce exposure to stolen or replayed publish credentials. For Hum, that means the adopter ask in 2026 is no longer “do you have checksums?” It is “can I verify this release back to a source tree, a builder identity, and an auditable workflow, and can I do that in my own environment?” **Confidence: high.** cite
