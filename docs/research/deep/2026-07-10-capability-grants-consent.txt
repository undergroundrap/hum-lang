# Capability Grants and Operator Consent for Hum

## Bottom line for decision 0017

**Finding:** Hum’s proposed rule — *runtime authority equals the program root’s declared capabilities intersected with the operator’s exact grant; default empty; deny wins* — matches the strongest patterns from object-capability systems, capability-routed OSes, WASI-style embeddings, and modern permission UX. The evidence is strongest for four subclaims: default-deny scales better than ambient authority, explicit attenuation beats broad roots, exact grants outperform wildcards for high-risk actions, and deny-precedence simplifies reasoning under composition. **Confidence: High.** cite

**(a) Research signals.** Deno’s runtime is sandboxed by default, exposes explicit `--allow-*` and `--deny-*` flags, and states that deny flags take precedence over allow flags. Fuchsia components only receive capabilities that are declared in manifests and successfully routed into the component namespace. WASI distinguishes *link-time capabilities* from *runtime capability handles*, both explicitly granted rather than ambient. Wasmtime’s `preopened_dir` API makes the grant tuple concrete: host path, guest path, directory permissions, and file permissions. Pony’s authority story is attenuative by construction: `AmbientAuth` can be downgraded into narrower tokens such as `TCPConnectAuth`, and the tutorial explicitly warns not to hand `AmbientAuth` to untrusted code. seL4’s capDL goes even further by making capability distribution declarative enough to reason about what future states of the system are possible. cite

**(b) Lessons for a new language.** For Hum, capability design should treat the *declared root set* as a static upper bound, the *operator grant* as a dynamic lower bound, and the *effective authority* as their intersection. That avoids ambient authority, makes least-authority derivation tractable, and gives security teams a simple algebra to explain. The important subtlety is that this algebra should be *typed and exact*, not just string-based: the strongest systems express authority as routable handles, narrowed tokens, or manifest entries rather than one giant “filesystem” or “network” bit. **Inference:** if Hum keeps the algebra exact and typed, its contracts/runtime blame system can point to the code path that *needed* a capability, while the capability algebra answers whether the task *had* authority to use it. cite

**(c) Concrete design gates or tests to adopt.** Gate Hum changes on four tests: first, a *monotonicity test* proving that adding operator-denies can never increase authority and adding undeclared app needs can never be satisfied at runtime; second, a *specificity test* showing that a grant to `fs.read("project/src")` does not imply `fs.read("~")`; third, a *route-completeness test* analogous to Fuchsia/seL4, so every granted capability has a traceable source and scope; and fourth, a *sandbox-bypass test* that treats capabilities such as subprocess launch, FFI, or arbitrary import as privileged “escape hatches” rather than ordinary grants. Deno’s own docs explicitly warn that `--allow-run` is effectively close to `--allow-all`, which is exactly the category Hum should segregate. cite

**(d) What evidence would change the conclusion.** I would soften this conclusion if there were strong field evidence showing that high-risk operator workflows are materially safer or more comprehensible with wildcard- or role-heavy grants than with exact, intersected grants. In the material reviewed, the opposite trend dominates: platforms keep moving from coarse, broad, or ambient authority toward narrower, contextual, and revocable grants. For Deno specifically, public evidence on what users *actually* grant in aggregate is **thin evidence / unknown**; the official sources describe the mechanisms well, but I did not find primary-source public telemetry comparable to Chrome’s permission studies. cite

## Grant algebra in real capability systems

**Finding:** The most reusable grant-algebra pattern is “broad roots exist only to mint narrower capabilities; routine code receives only the narrow form.” The systems differ in syntax, but they converge on attenuation, explicit routing, and first-class handles. **Confidence: High.** cite

**(a) Research signals.** Pony is the clearest language-level example: `AmbientAuth` can derive `NetAuth`, `TCPAuth`, and finally `TCPConnectAuth`, and the tutorial says library authors should request the *most specific authority* they need, not the ambient root. WASI documents a split between link-time capabilities and runtime handles; runtime handles are unforgeable and can be passed around without exposing unrelated resources. The Component Model’s WIT reference formalizes resources as owned or borrowed handles, which is a close match to Hum’s ownership- and view-oriented direction. Wasmtime then maps this to host reality with explicit preopens and separate directory/file permissions. Fuchsia adds a routing algebra over components with `use`, `offer`, `expose`, `capabilities`, and `availability`; critically, it forbids “upgrading” availability along a route. seL4/capDL uses declarative capability distribution not just for bootstrapping but for reasoning about what future states are reachable. Tahoe-LAFS demonstrates a different but very practical algebra: distinct read, write, and verify capabilities, plus “transitive readonlyness” so that read-only sharing does not silently smuggle write authority. cite

**(b) Lessons for a new language.** Hum should adopt a capability algebra with at least four first-class dimensions: **kind** of authority, **object or scope** the authority applies to, **strength** of the authority, and **lifetime/persistence** of the authority. Pony and Tahoe-LAFS show why “strength” must be explicit: read is not write, verify is not read, connect is not listen. WASI and the Component Model show why capabilities should be representable both as *static imports* and as *runtime handles*. Fuchsia and seL4 show why grants should also be *routable* or *explainable* across components/tasks, not merely attached to a process. The design implication for Hum is that the grant algebra should be compositional enough to survive tasks, workers, tools, and library boundaries without collapsing into a single ambient token. cite

**(c) Concrete design gates or tests to adopt.** Hum should require that every capability type can answer these questions mechanically: *What narrower capabilities can be derived from this one? What broader capability, if any, could have minted it? Can it be borrowed without transfer? Can it be persisted? Can it be routed across a task boundary without widening?* A good red-team test is to attempt “authority laundering”: start with a grant intended only for one task, pass it through helper APIs, and verify that no intermediate layer can widen or relabel it. Another is a “Tahoe test”: can a grant set contain a *verification* or *inspection* capability that is strong enough for integrity checks but too weak to disclose contents or mutate state? cite

**(d) What evidence would change the conclusion.** Evidence that a much simpler algebra — for example, just booleans per subsystem — produces materially better security/usability outcomes in language runtimes with real tools would weaken this recommendation. I did not find such evidence in the reviewed systems. The thin spot is E itself: the historical E materials strongly support capability discipline and leak avoidance, but they are older and the web-accessible primary-source detail is less complete than for Pony, Fuchsia, Deno, or WASI. That makes E a useful ancestor here, but not the main contemporary decision driver. cite

## Exact grants, wildcards, and persistence

**Finding:** Exact, task-scoped, and revocable grants outperform persistent wildcards for operator consent. Modern systems increasingly add *temporary*, *partial*, or *project-local* authorization modes after learning that broad persistent grants are sticky, hard to audit, and easy to regret. **Confidence: High.** cite

**(a) Research signals.** Android moved from coarse install-time permissions toward finer runtime controls such as “Only this time,” “Allow only while using the app,” “Ask every time,” repeated-denial suppression, auto-reset for unused apps, and selected-photos access rather than full media-library access. Chrome extensions likewise moved toward site-scoped and temporary authority: users can run an extension on click, on specific sites, or on all requested sites, and `activeTab` is a temporary user-gesture grant. Deno shows a similar evolution on the CLI/runtime side: permission descriptors can be exact to path/host/command/kind; `deno.json` supports `allow`, `deny`, and `ignore`; runtime prompts gained a new `A` option to grant all subsequent requests *within that domain*; and Deno now supports an external permission broker, including Windows named pipes, for centralized policy decisions. Tahoe-LAFS is the cautionary counterexample on persistence: capability URIs are durable bearer grants, and Tahoe’s own FAQ says the web UI does not have built-in authorization beyond secret object caps. cite

**(b) Lessons for a new language.** Hum should make *exactness* the default unit of consent and *persistence* an extra step, not the default. In practice that means operator grants should prefer `read:/project/data/report.csv` over `read:/project`, and `net:api.example.com:443` over `net:*`. If Hum permits wildcards at all, they should be visibly more dangerous, more reviewable, and harder to persist. Android and Chrome both show why: users understand “this time,” “while using,” “specific site,” and “selected photos” much better than hidden durable broad grants. Deno’s project-local config is a better persistence model than silent cross-project permanence, especially for a Windows-first language where multiple tools may coexist in the same machine context. Tahoe-LAFS, meanwhile, is a reminder that shareable bearer strings are powerful and sometimes desirable, but once you choose that model you must own the consequences: possession is authorization. cite

**(c) Concrete design gates or tests to adopt.** Hum should pass a *persistence review gate* before any grant can be saved: the runtime should show the exact capability tuple, persistence scope, expiration, and revocation path, and require explicit operator confirmation distinct from the one-shot allow action. It should also pass a *wildcard pressure test*: if a library or tool only works when the operator upgrades from exact grants to a wildcard, that is a release blocker unless the authority is truly inseparable. Finally, add an *auto-expiry test* for unused saved grants, mirroring Android’s reset logic, and a *partial-access test* for media/files/network so that tools are forced to prove they can degrade gracefully under narrower grants. cite

**(d) What evidence would change the conclusion.** Evidence from real operator studies showing that exact saved grants create intolerable friction in developer workflows — and that wildcard persistence does not materially raise overgrant or post-incident ambiguity — would change this recommendation. I did not find that evidence. For Deno specifically, public operator-behavior telemetry remains **unknown / thin evidence**; what is public shows the permission model’s mechanics and a real 2024 complaint that a broad subprocess semantic pushed one user off fine-grained permissions, but not aggregate grant behavior. cite

## Consent UX and the “just say yes” failure mode

**Finding:** Capability models fail in practice when they ask for consent out of context, too often, or in bundles that do not map to the user’s current task. The strongest evidence here comes from Android and Chrome: prompt timing and granularity dominate user behavior more than abstract security wording. **Confidence: High.** cite

**(a) Research signals.** Felt et al.’s classic Android study found only 17% of participants paid attention to permissions during installation, and only 3% of internet-survey respondents correctly answered all three comprehension questions, under an all-or-nothing install-time model. Their companion “How to Ask For Permission” paper argued for in-context requests tied to user goals. Chrome/web.dev’s 2024 guidance, based on Chrome usage statistics and user research, reports that 77% of desktop permission prompts are shown without a signal of user intent and only 12% of those are allowed; after a user interaction, allow rates rise to 30%. Chrome’s permissions-chip rollout was motivated in part by an approximately 85% ignore/dismiss rate for traditional prompts, and Chrome later added quieter prompts, automatic revocation for abusive notification sites, and 2026 push rate limits for low-value notification senders. Harbach’s 2024 Chrome study, using telemetry from more than 100 million Chrome installations and surveys from 25,706 users, found geolocation and notification prompts were most often ignored or dismissed and that perceived contextual information from the requesting site correlated with allow decisions. A 2025 grounded-theory study on mobile privacy pop-ups adds that *trust* and *privacy fatigue* emerge as dual motivations driving consent, which is exactly the mechanism that turns repeated prompts into “just say yes” behavior. cite

**(b) Lessons for a new language.** Hum should assume that repeated low-context prompts will train operators into either blanket-allowing or installing a broker/policy that suppresses prompts entirely. Therefore, operator consent should be *task-coupled*, *previewed*, and *recoverable*. Task-coupled means the request happens immediately next to the action the operator initiated, not at root startup and not lazily after unrelated computation. Previewed means the operator sees what will be accessed and why before the system-level grant surface appears. Recoverable means denial should have a graceful path forward, such as a fallback file picker, a narrower re-request, or a simple policy-edit surface. Deno’s evolution is informative but mixed: it moved from hard failures to interactive prompts, then added “allow all subsequent calls,” then stack traces for permission prompts, then a broker for higher-level policy mediation. That progression is useful evidence that developer-facing provenance matters, but it also hints that naive prompting is not sufficient on its own. cite

**(c) Concrete design gates or tests to adopt.** Hum should adopt a *no-startup-prompt rule* for nonessential capabilities; a *pre-prompt rule* that requires the language/runtime/tooling to present a plain-language rationale tied to one user action before any operator grant is requested; a *decision-quality metric* in usability testing that measures not just allow rates but later regret and policy comprehension; and a *fatigue budget* that caps prompts per session unless the operator explicitly enters a policy-edit mode. Also add a *developer provenance panel* showing the requesting package, task, source location, and call chain, because Android data-access auditing and Deno stack traces both suggest that provenance is essential when third-party libraries or SDK-like dependencies are involved. cite

**(d) What evidence would change the conclusion.** If controlled studies on code-execution tools showed that operators reliably make high-quality security decisions from frequent inline prompts without fatigue or blanket behavior, I would relax this recommendation. The reviewed evidence does not support that. If anything, both browser and mobile results suggest Hum should treat operator prompts as a scarce resource and spend them only when the user’s intent is already legible. cite

## Audit evidence for reconstructing who authorized what

**Finding:** Enforcement alone is not enough; Hum needs a grant log format that joins *policy snapshot*, *decision event*, and *runtime use event*. The best concrete templates in the reviewed systems come from Deno’s permission audit/broker schemas and Android’s data-access auditing. **Confidence: High.** cite

**(a) Research signals.** Deno can emit permission audits as JSONL or OpenTelemetry logs; its docs specify fields including format version, RFC 3339 timestamp, permission name, value, and optionally stack frames when tracing is enabled. Its broker request schema adds process ID, unique request ID, timestamp, permission, and value; the response schema adds the matching request ID, allow/deny decision, and optional denial reason. Deno also fails closed if broker messages are malformed or out of order. Android 11+ data-access auditing can log which operation was performed, distinguish app code from dependent SDK/library access on the requesting thread, include stack traces, and include attribution tags that map data access back to logical features of the app. Fuchsia manifests and seL4/capDL are not operator-consent logs, but they are valuable because they record the *intended route/topology* of capability availability, which is exactly the reference state an incident responder needs when comparing “what was possible by design” with “what was granted and used in practice.” cite

**(b) Lessons for a new language.** Hum’s grant log should record, at minimum, the operator identity or local authority surface that made the decision; device/session/process/task identifiers; the app root manifest hash or signed capability declaration; the exact requested capability tuple; the exact effective decision after intersection and denies; whether the decision was one-shot, session, project, or persistent; the UX surface and rationale text shown to the operator; the requesting package/library/source location; the timestamp; and a stable request ID that links the decision to later use events. **Inference:** for a security team to reconstruct “who authorized what,” you need both *authorization events* and *subsequent exercise events*. Deno gives a good model for authorization events; Android shows why exercise events need feature attribution and stack context; Fuchsia/seL4 show why you also need the declarative policy snapshot. Without all three, an investigation will be forced to guess intent from partial traces. cite

**(c) Concrete design gates or tests to adopt.** Hum should refuse to ship a capability subsystem unless it passes a *forensic replay test*: given only the logs and the signed root declaration, a reviewer must be able to answer which operator or broker authorized a capability, under what scope, for which task, from which code path, and whether the granted capability was later exercised outside the original task context. It should also pass a *tamper-evidence test* for persisted grants and decision logs, and a *dependency-attribution test* proving that accesses triggered by tools or libraries are attributed to the initiating package/task rather than disappearing into a generic runtime entry. cite

**(d) What evidence would change the conclusion.** If there were strong incident-response evidence showing that teams can reliably reconstruct capability decisions without per-request IDs, provenance, or policy snapshots, the required log shape could be smaller. I did not find such evidence. The opposite direction is more plausible: as Hum becomes more evidence-native and tool-rich, it may need *more* metadata than Deno or Android currently provide, especially around task lineage and contract blame. That last point is an inference from Hum’s design goals rather than a sourced industry standard. cite

## What capability systems got wrong first

**Finding:** The recurring first mistake was not “lack of capabilities” but *over-broad initial authority paired with consent surfaces too coarse to control it safely*. Systems then spent years adding attenuation, narrower scopes, temporary access, revocation, diagnostics, and auditability. **Confidence: Medium-High.** cite

**(a) Research signals.** Pony explicitly warns never to hand `AmbientAuth` to code you do not trust completely and instead derive the narrowest token needed. Deno’s docs warn that `--allow-run` is effectively close to `--allow-all`, and a 2024 user issue complained that subprocess semantics forced abandonment of fine-grained permissions in a real workflow. Android’s current security checklist says developers should minimize permission requests and, where possible, prefer access controls other than user-confirmed permissions because permissions can be confusing for users; it also warns against permission re-delegation. Felt et al.’s Android study showed how badly the initial install-time all-or-nothing model performed. Tahoe-LAFS shows another early sharp edge: its web UI has no built-in authorization beyond possession of secret caps, which is elegant for bearer-security but weak for operator-centric administration and auditing. Chrome extensions had to add runtime host controls so users could cut “all sites” down to “on click” or “specific sites.” cite

**(b) Lessons for a new language.** For Hum, the ruling should be: **do not ship broad roots and then hope UX or documentation will save you later**. If a capability is broad enough to bypass the model — subprocess, FFI, unrestricted imports, unrestricted filesystem, unrestricted network — it should be visually and semantically separated from ordinary capabilities. If a tool cannot operate under exact grants, that is evidence about the tool, not a reason to weaken the operator-consent model. The practical pattern visible across Android, Chrome, Deno, and Pony is that systems become safer when they move authority closer to the operator’s current intent and farther from ambient roots, persistent wildcards, and hidden delegation. cite

**(c) Concrete design gates or tests to adopt.** For decision 0017 this week, I would adopt the ruling as follows: keep **default-empty**, keep **deny-wins**, keep **effective authority = declared ∩ granted**, make grants **exact by default**, make persistence **explicit and reviewable**, and classify **sandbox-bypass authorities** as a separate severity tier with stronger warnings, shorter lifetimes, and mandatory audit logging. Add a release blocker that forbids any standard-library or tool API from requiring a broader authority than its implementation actually needs; Pony’s tutorial is explicit that libraries asking for more authority than needed should not be used. cite

**(d) What evidence would change the conclusion.** A credible contrary case would require recent, primary-source field data showing that operators using code-execution tooling are both more productive and not materially less secure under broad persistent grants than under exact, intersected grants. I did not find that. The strongest open uncertainty is not the correctness of the algebra, but the operational ergonomics: Hum will need user testing to decide how exact grant dialogs, persistence affordances, and broker/editor flows should look on Windows. That is a usability unknown, not a reason to weaken the security model. cite
