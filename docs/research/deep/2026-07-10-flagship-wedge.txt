# Flagship Wedge for Hum

## Recommendation

**Finding:** The best flagship wedge for Hum is an **offline air-gapped update validator**: a small Windows-first CLI that validates a release bundle from removable media or a staging share, checks hashes and signatures, optionally verifies Sigstore or in-toto provenance material, and emits a self-contained evidence bundle explaining exactly what was checked, by which policy, with which trusted roots, and why the verdict passed or failed. **Confidence: High.** This is the strongest fit because provenance and auditability are already first-order concerns in software acquisition and supply-chain guidance; offline verification is a real operational need in air-gapped environments; and today’s tooling is powerful but fragmented across Sigstore/Cosign, TUF, in-toto, Ratify, CycloneDX, and SBOM analyzers rather than delivered as one operator-grade “accept or reject this update” utility. CISA’s acquisition guidance explicitly asks whether suppliers obtain and manage provenance information such as SBOMs, while NIST’s current traceability work emphasizes capturing the minimum data needed to verify provenance or compliance even when supplemental links are unavailable. NIST’s platform-integrity guidance also documents concrete offline or air-gapped verification workflows. cite

**Finding:** My ranked top five candidates are: **air-gapped update validator**; **Windows config drift auditor**; **SBOM and provenance verifier**; **offline secret scanner**; and **deterministic log scrubber/redactor**. **Confidence: Medium-High.** The ranking reflects three filters: whether provenance is the main reason to choose the tool; whether there is a meaningful gap between existing frameworks and a small operator utility; and whether a compelling v1 could plausibly be built in roughly 2,000 lines if Hum leans on existing file formats and cryptographic libraries instead of rebuilding an ecosystem. The main reason I place the Windows drift auditor above the SBOM verifier is strategic, not purely market-size: Microsoft’s 2025-2026 OSConfig and DSC work makes baseline verification newly concrete on Windows, which matches Hum’s Windows-first positioning. By contrast, the SBOM/provenance verifier space is crowded with generators, scanners, and graph systems already. cite

**(a) Research signals.** The strongest 2024-2026 signals all point toward evidence-rich verification, not merely scanning: Sigstore bundles package the material needed to verify signatures; Sigstore’s Go client now explicitly supports offline verification, custom trusted roots, and structured verification results; CISA’s 2024 SBOM framing raises expectations for author identity, timestamps, and baseline attributes; and CycloneDX Attestations positions claims plus evidence as machine-readable compliance artifacts. Those are exactly the building blocks that make an “evidence bundle” demo legible to buyers. cite

**(b) Lessons for a new language.** The wedge should not be “a better scanner.” It should be a **better receipt**: one command in, one verdict plus one dossier out. The winning interaction is, “Here is `update.zip`; here is `update.evidence.json`; here is why it is safe enough to import into a disconnected environment.” That directly showcases Hum’s stated strengths—runtime-checked contracts, capability-bounded effects, and evidence-native tooling—without asking users to buy into a new language first.

**(c) Concrete design gates or tests to adopt.** A Hum v1 should pass four gates before public demo: first, **zero-network operation** on a fresh Windows machine; second, **deterministic evidence output** on repeated runs over the same bundle; third, **policy explainability**, where every failure names the exact missing artifact, signer, threshold, or claim; and fourth, **effect minimalism**, where the emitted evidence itself shows that only declared read/write capabilities were exercised. Those gates are product requirements, not just engineering niceties. They mirror the offline-verification and structured-result expectations already visible in current supply-chain tooling. cite

**(d) What evidence would change the conclusion.** I would revise the ranking if you found two pieces of contrary evidence: first, strong user interviews showing that buyers care more about **continuous monitoring** than **acceptance-time provenance**; second, hard evidence that security teams in Hum’s initial target market already have a simple, trusted, offline acceptance tool they genuinely like. I did **not** find that second thing. Instead, I found frameworks, registries, scanners, and admission controllers. That is a gap, not a solved UX.  

## Air-gapped update validator

**Finding:** This is the best wedge because it turns existing supply-chain primitives into a concrete operator action: **accept or reject a software drop in an offline enclave**. **Confidence: High.** Today’s ingredients are mature. TUF defines threshold-signed metadata and explicit trusted roles; in-toto verifies that a supply chain matches a signed layout and checks thresholds and artifact rules; Cosign verifies blobs using a bundle that contains signature and certificate material; Sigstore’s bundle format states that a bundle is everything required to verify a signature on an artifact; and Sigstore’s Go client supports online and offline verification with custom trusted roots. Ratify shows that policy-based verification of artifact-security metadata is valuable, but it is positioned as a framework or engine, especially around OCI and Kubernetes, not as a tiny Windows acceptance utility. NIST’s 1800-34 guidance is especially persuasive here because it describes actual offline or air-gapped integrity verification flows. cite

**Existing tools and known gaps.** The gap is not “verification is impossible.” The gap is **operator-grade composition**. Cosign can verify blobs and bundles, but Sigstore’s own integration guidance says Cosign is a legacy system for integration, recommends Sigstore-go for verification, notes no API stability guarantees for Cosign integration, and says Cosign lacks support for the Protobufs-based bundle format. TUF and in-toto are strong frameworks, but they ask users to understand roles, thresholds, layouts, link metadata, and trust roots. Ratify is a verification engine, but that is still heavier-weight and more platform-oriented than a single Windows executable a defense team can run against `\\staging\drop42\`. Minisign and signify are admirably small, but they verify integrity and signatures rather than higher-level provenance claims. **That** is the opening for an evidence-native tool: unify the checks and emit one dossier. cite

**Why evidence-native would beat incumbents.** Hum’s edge would not be a new signing scheme. It would be the output: a **portable evidence bundle** containing artifact digests, trusted-root identifiers, signer identities, threshold status, contract results, optional SBOM/VEX references, and a machine-readable pass/fail explanation. That maps cleanly onto current provenance and attestation work: CISA’s buyer guidance asks for provenance information; NIST’s traceability draft stresses enough embedded data to validate pedigree even when links are inaccessible; and CycloneDX Attestations explicitly frames compliance claims as evidence-backed, machine-readable artifacts. Existing tools verify pieces; the winning demo is one tool that **proves what it checked**. cite

**Realistic scope for a ~2000-line implementation.** The feasible v1 is narrower than “full Sigstore/TUF client.” A realistic v1 would accept a manifest plus payload directory, verify SHA-256 digests, verify one or more detached signatures or Sigstore bundles, enforce a small local policy file, and emit a JSON evidence bundle. If you support a limited set of trust models—say, Ed25519 public keys, signed checksum lists, and one Sigstore-bundle path—you can ship something useful without rebuilding the ecosystem. `signify -C` already proves that signed checksum-list verification is a practical pattern, and Cosign’s `verify-blob --bundle` proves that a bundle-driven blob workflow is viable. cite

**Likely first ten users.** This is an **inference, not a sourced customer list**. The first ten user archetypes are likely to be: an IT administrator for a disconnected lab; a defense integrator validating vendor drops from removable media; an OT/ICS security engineer importing patches into a restricted network; a Windows desktop-engineering team validating internal package drops; a security-compliance team that must retain audit artifacts; a platform-security engineer at a contractor subject to software-acquisition questionnaires; a software factory release manager handing artifacts into higher-classification environments; a blue-team tooling engineer at a federal agency; a supplier-assurance analyst validating third-party binaries against declared provenance; and a security-conscious open-source project that wants a tiny verifier for release ZIPs. The basis for this inference is CISA’s software-acquisition emphasis on provenance and NIST’s documented offline validation scenarios. cite

**(a) Research signals.** The strongest signals are the convergence of Sigstore bundles, TUF thresholds, in-toto verification, Ratify policy enforcement, and NIST’s air-gapped integrity examples. cite

**(b) Lessons for a new language.** Keep the trust story local, visible, and boring. Too much of current supply-chain security is conceptually right but operationally abstract. The language demo should make trust roots, identities, thresholds, and failed predicates visible in plain English, then show the exact evidence bundle produced.

**(c) Concrete design gates or tests to adopt.** Require: offline-only verification on Windows; a corpus of good and tampered bundles; signer-threshold tests; expired-certificate and bad-issuer cases; and golden-file tests for the emitted evidence JSON. Also add a “link rot” test where supplemental references are absent, because NIST explicitly warns that provenance validation must survive unavailable supplemental links. cite

**(d) What evidence would change the conclusion.** If interviews reveal that offline buyers mostly want **package transport** or **media sanitation** rather than **artifact acceptance**, I would narrow or redirect this wedge. Likewise, if real operators reject provenance bundles and only care about a green/red status bit, the evidence-native differentiated output may be overbuilt. I currently see **thin evidence** for that simpler preference.  

## Windows config drift auditor

**Finding:** If Hum wants a more explicitly Windows-native wedge, the runner-up is a **Windows config drift auditor** that reads a signed baseline, checks local state, and emits a signed evidence report describing drift at the file, registry, service, and policy level. **Confidence: Medium-High.** Microsoft now has a much clearer 2025-2026 story around OSConfig for Windows Server 2025 and DSC v3. OSConfig is described as a security-configuration tool that applies and maintains a known-good state and detects and corrects drift; its baseline docs explicitly support offline installation; DSC v3 is declarative, cross-platform, has no external dependencies, and its `dsc config test` output includes per-resource desired state, actual state, changed properties, messages, and metadata. InSpec remains a widely used compliance-as-code auditor with JSON reporting. cite

**Existing tools and known gaps.** The existing market is full of frameworks and enterprise pipelines, but the small-tool gap is still real. OSConfig, DSC, and InSpec all produce useful outputs, yet the output formats are not natively framed as **portable audit evidence dossiers** that also capture declared effects, trust roots, and a deterministic verification story. OSConfig gives compliance status and reasons; DSC returns structured `metadata`, `results`, `messages`, and `hadErrors`; InSpec can emit JSON-based reports. What is missing is a tiny Windows-first verifier whose default artifact is not “console output” or “a JSON report” but “a chainable, signed evidence object someone can file with change-control or an auditor.” That distinction is subtle but strategically important for Hum. cite

**Why evidence-native would beat incumbents.** Hum could turn configuration checking from “a scan happened” into “this exact baseline was applied as a contract, these resources were read, these ones were modified or left untouched, and here is the caller/task blame for every violation.” That is especially compelling in change-controlled Windows environments because the baseline itself, the compliance verdict, and the allowed effects can all travel together. I have **thin evidence** that mainstream buyers explicitly ask for this packaging today, but the underlying demand for known-good state, drift control, and compliance reporting is well documented in Microsoft’s current guidance. cite

**Realistic scope for a ~2000-line implementation.** A feasible v1 is not “Hum replaces DSC.” It is “Hum reads a minimal baseline schema and checks a narrow slice of Windows state.” Pick perhaps four resource kinds: file hash or presence, registry value, service startup state, and installed module or executable version. Emit one evidence bundle that includes baseline ID, machine ID, pass/fail per check, and an overall digest. Because DSC already demonstrates that checking desired versus actual state yields compact structured results, a Hum tool can stay small if it intentionally limits resource coverage. cite

**Likely first ten users.** This is **inference**. Likely first adopters are: Windows Server administrators in regulated enterprises; Azure Arc or OSConfig operators wanting offline audit artifacts; red-team or purple-team labs wanting pre/post hardening receipts; IT auditors validating server baselines; MSSPs delivering Windows hardening reports; contractors managing secure member-server images; domain-admin teams validating domain-controller drift; desktop-engineering teams managing golden images; SOC engineers triaging “what changed” during suspected compromise; and change-advisory boards that want attachable evidence rather than screenshots or ad hoc PowerShell transcripts. The inference is grounded in Microsoft’s current emphasis on drift control, security baselines, and compliance details. cite

**(a) Research signals.** OSConfig and DSC are maturing fast, and they already normalize the idea of baseline documents, test results, and structured messages. cite

**(b) Lessons for a new language.** A strong Hum demo here would not argue that Microsoft lacks managed-state tooling. It would argue that **evidence quality and operational portability** are still weak at the small-tool layer.

**(c) Concrete design gates or tests to adopt.** Require reproducible results on repeated scans; schema validation for baseline inputs; explicit enumeration of all reads and writes in the evidence output; and side-by-side comparisons against DSC on a small resource corpus. Use at least one offline-install test, since Microsoft documents that operational mode directly. cite

**(d) What evidence would change the conclusion.** If user testing shows that Windows administrators mostly want **automatic remediation** and not **auditable verification**, this wedge drops in priority. If Hum cannot deliver a first-class Windows UX, it also drops.  

## SBOM and provenance verifier

**Finding:** A narrow **SBOM and provenance verifier** is a good candidate, but only if it is tightly scoped to **verification and explanation**, not generation, graphing, vulnerability management, or registry automation. **Confidence: Medium.** The current landscape is active: Syft generates SBOMs and can produce signed in-toto SBOM attestations; CycloneDX documents authenticity verification and lists `cyclonedx-verify` as a CLI that checks integrity against CycloneDX SBOMs; Trivy can scan SBOMs in CycloneDX, SPDX, and CycloneDX-type attestation formats; GUAC ingests SBOMs, SLSA attestations, VEX, and related metadata into a graph; and CISA’s 2024 SBOM framing raises baseline expectations for author identity, timestamps, and other attributes. cite

**Existing tools and known gaps.** The problem is not absence of tooling but **fragmentation and trust ambiguity**. Trivy itself warns that SBOMs generated by tools other than Trivy may produce inaccurate detection because Trivy relies on custom properties for accurate scanning. Syft plus Cosign plus Grype form a viable chain, but that is still multiple tools. GUAC is strong for aggregation and query, but it is not a tiny verifier. CISA’s framing is explicit that minimum attributes and maturity levels matter, which means a verifier that tells users not just “this SBOM parses” but “this SBOM is too weak for your policy because the author identity, timestamp, or component identity quality is insufficient” would create more value than another SBOM generator. cite

**Why evidence-native would beat incumbents.** A Hum verifier could accept an SBOM plus optional attestation bundle and emit a signed, machine-readable verdict explaining: schema validity, integrity verification, provenance status, policy sufficiency, and any missing evidence. CycloneDX Attestations explicitly frames claims as evidence-backed compliance artifacts, so a Hum evidence bundle would fit the direction of travel instead of inventing a new concept. Still, I rank this below the air-gapped validator because the demo is less visceral unless attached to a concrete artifact-acceptance workflow. cite

**Realistic scope for a ~2000-line implementation.** Feasible v1: parse CycloneDX JSON only, verify attached or companion signature material, require a minimum policy such as author identity plus timestamp plus root component plus hashes, and emit a verdict bundle. Not feasible in 2,000 lines: multi-format full fidelity, vulnerability correlation, registry discovery, or graph expansion. Trivy, Syft, and GUAC already show how quickly scope explodes once you leave pure verification. cite

**Likely first ten users.** **Inference.** Likely early users are: supplier-assurance teams receiving third-party SBOMs; product-security engineers validating release evidence for major customers; CI teams that want a post-build “receipt” step; compliance engineers mapping SBOM maturity to buyer questionnaires; security architects piloting SLSA-style controls; open-source maintainers shipping signed SBOMs; medical or industrial vendors under component-transparency pressure; internal audit teams checking whether an SBOM is policy-sufficient; risk teams reviewing externally supplied binaries; and SBOM-curious teams who find GUAC or Dependency-Track too big for the first step. The support here comes from CISA’s SBOM framing and the growing tool ecosystem around attestations and supply-chain metadata. cite

**(a) Research signals.** SBOM verification is becoming more operational, but the ecosystem remains generator-heavy and platform-heavy. cite

**(b) Lessons for a new language.** Verification UX matters more than feature breadth. “Your SBOM is valid but insufficient for policy X” is more useful than “SBOM parsed successfully.”

**(c) Concrete design gates or tests to adopt.** Require differential tests on good, malformed, weak, unsigned, and mismatched SBOMs; produce deterministic evidence bundles; and compare your verdicts against `cyclonedx-verify`, Syft attestations, and Trivy ingestion on a tiny corpus. cite

**(d) What evidence would change the conclusion.** If interviews show that customers primarily want **vulnerability triage** rather than **evidence sufficiency**, this candidate should move down and stay a feature inside the air-gapped validator instead of becoming the wedge itself.  

## Offline secret scanner

**Finding:** An **offline secret scanner** is attractive and easy to explain, but provenance is not yet the dominant differentiator in this category; accuracy and triage quality still dominate. **Confidence: Medium-Low.** The current field is crowded. Gitleaks scans repos, directories, and stdin, supports baselines, and supports allowlisting. `detect-secrets` is explicitly baseline-centered and provides audit workflows to tune signal-to-noise. TruffleHog classifies more than 800 secret types and emphasizes automatic live verification of credentials against provider APIs, including filesystem and stdin sources. That makes it powerful, but it also reveals the category’s central tension: the best-known differentiator today is not provenance, but verified liveness. cite

**Existing tools and known gaps.** The gap is strongest in **restricted environments**. TruffleHog’s verification model depends on checking credentials against live services; that is precisely what an air-gapped or high-side environment may refuse to do. Gitleaks and detect-secrets work offline, but both document ongoing baseline, allowlist, and tuning workflows—which is a polite way of saying false-positive management is part of life in this category. So the Hum angle is plausible: promise a scanner that is offline by construction, effect-bounded, and able to emit a signed evidence bundle proving what sources were scanned, with which rulepack version, with which allowlists, and without network access. That is a differentiator, but it is still less immediate than the update-validator story. cite

**Why evidence-native would beat incumbents.** In high-trust or defense settings, proving **non-exfiltration** can matter almost as much as the findings themselves. A Hum scanner could make that claim concrete if its evidence bundle shows that the process used only directory-read and local-write capabilities and never exercised network effects. That is a good fit with the prompt’s capability-based-effects vision. The limitation is commercial, not technical: I found **thin evidence** that buyers currently select secret scanners primarily on auditable side-effect boundaries rather than on detector quality, live verification, or CI ergonomics.

**Realistic scope for a ~2000-line implementation.** Feasible v1: filesystem and stdin only, a narrow rule set, deterministic output, baselines, and allowlists. Not feasible in 2,000 lines: 800+ detector taxonomy, live validation against many providers, broad SaaS integrations, or full historical git archaeology comparable to mature tools. The strongest small-scope version would be a “USB and working-tree scrubber” for disconnected environments rather than a general-purpose replacement for Gitleaks or TruffleHog. cite

**Likely first ten users.** **Inference.** Likely early users are: security teams scanning removable media before import; red teams carrying offline tooling into labs; incident responders triaging exfiltration risk on disconnected images; regulated developers who need local-only secret checks; defense contractors scanning delivery folders; DevSecOps teams that distrust networked verification from sensitive environments; threat-hunting labs processing log bundles offline; secure-build teams scanning release artifacts before enclave transfer; Windows developers who want a dead-simple local preflight scanner; and compliance teams that need signed scan receipts. The inference rests on the documented offline capability of Gitleaks and detect-secrets and the network-oriented verification model of TruffleHog. cite

**(a) Research signals.** This is a real and active category, but the current winners compete mostly on detection breadth, verification, and triage. cite

**(b) Lessons for a new language.** If Hum goes here, the demo must be about **trust boundaries** and **audit receipts**, not “we scan secrets too.”

**(c) Concrete design gates or tests to adopt.** Require zero-network test harnesses, reproducible findings, baseline compatibility, and explicit proof in the evidence bundle that no network capability was exercised. Also build a fixture corpus with canary tokens, fake secrets, and known false positives. cite

**(d) What evidence would change the conclusion.** If interviews show that high-side teams desperately need **local-only secret scanning with receipt-quality evidence**, this candidate rises sharply. If not, it remains a secondary wedge or a feature in a broader acceptance tool.  

## Deterministic log scrubber and redactor

**Finding:** A **deterministic log scrubber/redactor** is technically well aligned with Hum’s contracts and effects, but it is the weakest of the top five as a wedge because provenance and auditability are usually secondary to detection quality and integration in this category. **Confidence: Low-Medium.** The space is active: OpenTelemetry documents transforming telemetry for governance, cost, and security reasons and recommends the Collector’s `redaction` processor for scrubbing sensitive data; Vector documents PII redaction and emphasizes operational guarantees; Microsoft Presidio provides PII detection and redaction tooling, including evaluation examples. Those are substantial incumbents, even if they are more pipeline- and SDK-oriented than tiny CLIs. cite

**Existing tools and known gaps.** The missing piece is **provable transformation history**. OTel and Vector can transform and redact, but they are primarily sold as data-processing components inside observability stacks. Presidio is a privacy toolkit, not a tiny operator utility. A Hum redactor could differentiate by producing an evidence bundle that records which transform rules were applied, which fields were dropped, hashed, or preserved, what false-negative or allowlist rules were active, and whether any undeclared effects occurred. I found **thin evidence** that this is a decisive purchase criterion today, which is why I rank it fifth even though the language fit is strong. cite

**Why evidence-native would beat incumbents.** The best case is environments where sanitized logs are handed from a high-trust side to a lower-trust side. In that workflow, the redaction result is not enough; the receiver wants a **receipt** for what was scrubbed. CycloneDX’s evidence-management use case is relevant by analogy here: auditors like structured evidence, not screenshots and tribal knowledge. Yet this is an analogy, not a direct sourced statement about log-redaction procurement, so treat it as **inference**. cite

**Realistic scope for a ~2000-line implementation.** Feasible v1: newline-delimited JSON logs only, deterministic field allowlist or denylist, regex-based token replacement, and evidence output summarizing transformed fields and counts. Not feasible in 2,000 lines: multilingual PII detection, model-based extraction, distributed pipeline integration, or image/PDF redaction parity with Presidio. A compact CLI could still be compelling if positioned as a **handoff sanitizer** rather than a general observability platform component. cite

**Likely first ten users.** **Inference.** Likely first users are: IR teams preparing logs for external sharing; contractors moving evidence across trust boundaries; SOC analysts needing safe bundles for vendors; legal/compliance teams sending sanitized incident records; SRE teams shipping support bundles to offline review stations; product-security teams sharing logs with developers under least-privilege constraints; Windows administrators exporting event logs for third-party analysis; privacy offices reviewing redaction procedures; healthcare or regulated environments sharing telemetry; and security researchers publishing sanitized repro artifacts. This inference is grounded in the current documentation around redaction for security and privacy, not in a direct buyer list. cite

**(a) Research signals.** Strong platform-level tooling exists, but the standalone evidence-first CLI niche is less clearly validated. cite

**(b) Lessons for a new language.** Hum would need to position this as a **controlled data-handoff tool**, not as a “log pipeline competitor.”

**(c) Concrete design gates or tests to adopt.** Require deterministic replacement behavior, explicit rule provenance in the output, no-network execution, and a before/after corpus that reviewers can inspect manually.

**(d) What evidence would change the conclusion.** If customer discovery finds repeated pain around **sanitizing logs for cross-boundary sharing** with auditable receipts, this wedge moves up. Otherwise it stays a useful follow-on utility, not the flagship.  

## Killer demo patterns

**Finding:** The “undeniable demo” pattern behind ripgrep, jq, and age is not magic; it is a repeatable product shape: **one sharp job, obvious before/after, composable output, and almost no setup**. **Confidence: Medium.** For ripgrep, the original announcement and current README made the case with concrete command lines, immediate speed comparisons, `.gitignore`-aware defaults, and first-class Windows support; the demo answers “why not grep?” in under a minute. For jq, the manual starts with the identity filter and field extraction, turning unreadable JSON into obviously useful output, and the README emphasizes that it is like `sed`/`awk`/`grep` for JSON with zero runtime dependencies. For age, the README’s minimal example—generate key, encrypt tar stream, decrypt tar stream—shows a complete mental model in three commands, while Filippo Valsorda’s writing makes the design philosophy explicit: cryptographic tools work best when specialized and opinionated rather than as flexible Swiss Army knives. cite

**Thin evidence note.** I did **not** find formal 2024-2026 maintainer post-mortems saying “this one demo caused adoption” for all three tools. The ripgrep announcement is effectively a launch argument, and jq’s manual and age’s README are living product demos rather than retrospective analyses. So the conclusions here are partly **inference from primary source material**, not direct maintainer testimony. **Confidence: Medium.** cite

**What made the demos undeniable.** Ripgrep’s demo was comparative and merciless: faster than familiar tools, smarter defaults, and no penalty for Windows users. jq’s demo was transformational: pipe in ugly JSON, get a readable answer immediately. age’s demo was trust-reducing: fewer knobs, explicit keys, obvious composition with shell tools, and a complete happy path that fits on screen. Across all three, the product story was not “here is an architecture.” It was “here is a command that solves a problem you already have.” cite

**(a) Research signals.** The primary sources point to four recurring traits: smart defaults, composability, minimal dependency burden, and a visible comparison against the incumbent. ripgrep advertises recursive search plus ignore semantics by default and publishes benchmark-style examples; jq centers filters and standard I/O; age centers tiny examples and explicitly rejects over-generalized crypto UX. cite

**(b) Lessons for a new language.** Hum’s wedge demo should therefore avoid “language introduction” framing. The demo should be: **drop a release bundle into a folder, run one Hum-built executable, and receive a verdict plus an evidence bundle that a skeptical reviewer can inspect without learning Hum**. The language becomes interesting only after the utility is undeniable.

**(c) Concrete design gates or tests to adopt.** The launch demo should pass seven tests. It should finish in under a minute on Windows. It should require no service account and no network. It should take a familiar artifact format like ZIP plus manifest. It should show a crisp failure mode on a tampered file. It should emit both human and machine-readable output. It should prove its own limited effects in the evidence report. And it should have an obvious incumbent comparison: for example, “Cosign verifies the blob, but here is the single dossier Hum produces for the whole drop.” The supporting precedent for “structured verification results” and “bundle as all required verification material” already exists in Sigstore. cite

**(d) What evidence would change the conclusion.** If early users respond better to a **Windows drift proof** than to a **software-drop acceptance proof**, pivot the same demo shape to the config drift auditor. The critical thing is not the exact utility; it is preserving the killer-demo pattern: one obvious job, one command, one legible result, and one evidence artifact that the incumbent does not naturally produce.
