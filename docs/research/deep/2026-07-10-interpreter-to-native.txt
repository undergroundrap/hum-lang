# Interpreter to Native for Hum

## Executive synthesis

**Finding:** The strongest evidence supports a **dual-track path** for Hum rather than a single backend bet: keep the interpreter as the semantic oracle, target **Wasm/WASI components early** for capability-disciplined packaging and embedding, and bring up **direct native codegen through Cranelift** as a second path for fast developer builds and selected native deployments. I would **not** recommend treating Cranelift as Hum’s only shipping native backend for a Windows-first language yet, because the Rust and Cranelift ecosystems themselves still describe Windows support, debug info, and some exception/unwind work as materially harder than Linux/macOS. This is an inference from the current state of Cranelift, `rustc_codegen_cranelift`, Wasmtime, and current WASI target support. **Confidence: High.** cite

**Research signals:** The Rust project’s 2025H2 goal is explicitly to make the Cranelift backend “production-ready” for **local development**, not as a universal drop-in replacement for LLVM, and it still names **Windows** as a harder, longer path and flags **debug info** as a major missing feature. Cranelift’s own README calls it production-ready **within Wasmtime contexts**, reports code quality roughly **2% slower than V8 TurboFan** and **14% slower than LLVM-based WAVM**, with compilation speed about **an order of magnitude faster than WAVM**. At the same time, the WASI/component stack has become much more real: the component-model repository says Preview **0.2** and **0.3** features are kept stable by producer and consumer tools for production feedback, Rust’s `wasm32-wasip2` target is now Tier 2 and supports `std`, and Wasmtime’s WASI builder is explicitly deny-by-default for many ambient capabilities. cite

**Lessons for a new language:** Hum should treat its interpreter as more than a bootstrap artifact. It should be the executable semantics, the blame oracle for `needs:`/`ensures:` and `old()`, and the differential-testing baseline for every compiled backend. The native story should then be staged: first prove semantic fidelity and diagnostics through the interpreter and Wasm/components; only then raise Cranelift to “day-to-day native default” status on the platforms where it demonstrably clears Hum’s gates. This is an inference, but it is directly encouraged by the way CPython, Ruby YJIT, and OCaml have constrained semantic drift while growing faster execution tiers. **Confidence: High.** cite

**Concrete design gates or tests to adopt:** Hum should not promote a backend merely because it can run “real programs today.” Promotion should require: semantic equivalence on a differential corpus; contract blame equivalence across interpreter and compiled backends; source-location fidelity through debugger-visible stepping and variable inspection; capability-denial tests with no ambient access by default; and quantified cost ceilings for retained runtime checks. A backend that fails any one of those should stay preview-only. cite

**What evidence would change the conclusion:** I would become substantially more positive on a Cranelift-first native route for Hum if three things happen together: first, shipping-quality **Windows unwind/SEH and debug-info support** in the Cranelift stack; second, public evidence that `rustc_codegen_cranelift` has crossed from “local dev preview” to stable recommendation on Windows; and third, a Hum-specific benchmark suite showing retained contracts and overflow traps stay within an agreed overhead budget on representative workloads. I would become less positive on Wasm/components as the early deployment target if the async/component story remains fragmented or if debugger fidelity for components continues to lag. cite

## Cranelift as a native backend in mid-2026

**Finding:** Cranelift is mature enough to be taken seriously for Hum’s native path, but the evidence fits a narrower statement than “ready to replace LLVM everywhere”: it is strongest for **Wasmtime and related Wasm execution**, credible for **fast native codegen in development workflows**, and still weaker for a **Windows-first, debugger-heavy systems language toolchain**. **Confidence: High.** cite

**Research signals:** Cranelift’s README says plainly that it is “production-ready” and used in production in several places, but it immediately scopes that claim to “all within the context of Wasmtime.” The same document says Cranelift has four backends—x86-64, AArch64, s390x, and riscv64—and supports the Windows x64 calling convention on x86-64. Its published third-party comparison says generated code is about **2% slower than V8 TurboFan** and about **14% slower than LLVM-based WAVM**, while compilation is about **10x faster than WAVM**. The Rust ecosystem is converging on the same positioning: the Rust project goal frames Cranelift as a path to faster local compilation, cites roughly **20% lower codegen time** and around **5% faster total clean builds** on large projects, and explicitly says the target state is “default for local development” rather than “universal release backend.” The same goals page says Windows is “much more challenging,” and names **debug info** as a major missing feature. The `rustc_codegen_cranelift` repository is available as a nightly rustup component on Linux, macOS, and **x86_64 Windows**, which is meaningful, but its maintainer’s June 2025 report still says unwinding support remains disabled by default and does **not** work on Windows. cite

There is also a useful negative signal from Wasmer. Wasmer’s runtime docs still offer Cranelift as one of its compiler backends and describe it as focused on compile time and runtime performance, and Wasmer 7.0 in January 2026 highlighted new exception support in Cranelift. But Wasmer’s own `wasmer-compiler-cranelift` documentation says to use Cranelift mainly for **development**, and recommends LLVM for production, claiming LLVM gives much better runtime speed in their setting. That does **not** prove a universal 50% LLVM advantage for all workloads; it is a Wasmer-specific recommendation and should be read that way. Still, it is a valuable signal that sophisticated embedders continue to see a meaningful performance/feature split between Cranelift and LLVM in 2026. **Confidence: Medium-High.** cite

**Lessons for a new language:** For Hum, Cranelift’s real advantage is not “ultimate code quality.” It is the combination of **fast compile latency**, **small integration surface**, **Rust-native embedding**, and enough code quality to make native execution practical early. That is exactly the profile that helps an interpreter-first language get into developer hands quickly. The evidence does **not** support assuming Cranelift will give Hum its best release-mode Windows experience in the near term, especially if Hum wants first-class PDB/debugger support, runtime blame source-fidelity, and robust exception/unwind behavior. That second sentence is an inference from the Rust goals page, the maintainer report, and the broader debug-info evidence below. **Confidence: High.** cite

**Concrete design gates or tests to adopt:** Hum should establish a “Cranelift promotion ladder.” The first rung is simple native correctness on Linux and macOS with a large differential corpus. The second rung is ABI validation across FFI boundaries, ideally using the sort of ABI-focused comparison discipline that the `rustc_codegen_cranelift` maintainer references with `abi-cafe`. The third rung is Windows-specific: unwind tables, exception propagation or trapping semantics, debugger stepping, local-variable inspection, and blame-site preservation for contract failures. The fourth rung is performance: Hum should require a measured speedup over its interpreter on a native corpus, and a cold-build improvement over any LLVM path large enough to justify the maintenance split. Only after those gates should Cranelift become Hum’s default native backend on a given platform. cite

**What evidence would change the conclusion:** A future Rust project-goals update that changes the Cranelift objective from “local development” to something closer to “general recommendation,” paired with public Windows debugger and unwind milestones, would materially strengthen the case for a Cranelift-first Hum native story. Conversely, if Wasmer- and Rust-like users keep recommending LLVM for production in 2026-2027, that would strengthen the case for Hum to maintain an LLVM escape hatch or at least a Wasm-first deployment route longer than originally planned. cite

## Wasm and WASI components as an early target

**Finding:** The Wasm/WASI component model is a very strong fit for Hum’s capability-based, deny-by-default design and should be taken seriously as an **early deployment and embedding target**, but it is still not mature enough to be Hum’s only backend story if Hum needs stable async, rich platform tooling, and a polished Windows-native debugger experience. **Confidence: High for capability alignment, Medium for “first target only.”** cite

**Research signals:** The component-model specification repository says the model is being standardized incrementally through WASI developer previews, and that features enabled in a WASI preview release are kept stable by producer and consumer tools so they can be used in production settings to gather real-world feedback. The same repository identifies **WASI 0.2** as the first release based on the component model, with linking, higher-level value types, `resource` handles, and WIT, and **WASI 0.3** as the first release with native concurrency support such as `async` functions, `stream`s, and `future`s. On the Rust side, the 2024 Rust blog announced `wasm32-wasip2` Tier 2 support in Rust 1.82, and the current target documentation says `wasm32-wasip2` supports `std` fully, outputs **components rather than core Wasm modules**, and expects runtimes with Preview 2 plus component support. That same target page still says the target is **not tested in CI at this time**, which matters. Meanwhile, Rust already documents a `wasm32-wasip3` target, but only at Tier 3, and its own docs note that WASIp3 had not yet been approved and that the target did not yet build as of the dated note embedded in the page. Wasmtime’s public API docs also still mark some component-model features as incomplete. cite

The capability story, by contrast, is unusually strong and direct. Wasmtime’s `WasiCtxBuilder` defaults to closed stdin, inert stdout/stderr, no env vars, no arguments, no preopens, and socket policies that deny addresses and deny IP name lookup by default. This is exactly the kind of no-ambient-authority stance Hum appears to want at the language level. The important point is that this is not a vague philosophical alignment; it is visible in the default host API surface of a major WASI runtime. **Confidence: High.** cite

**Lessons for a new language:** For Hum, a Wasm/component target can do two jobs at once. First, it is a deployment substrate with explicit imports and capability handles that naturally fit a deny-by-default language. Second, it can be a forcing function on Hum’s module and effect model: if a Hum package cannot state its imports cleanly in a component boundary, that is useful design feedback. The current evidence suggests this is a better **first distribution target** than a first-class substitute for native Windows binaries. Put differently: components are a good place for Hum to become precise about effects, resources, and authority; they are a weaker place to make promises about top-tier native debugging and platform integration. This is an inference from the standards, Rust targets, and Wasmtime runtime defaults. **Confidence: High.** cite

**Concrete design gates or tests to adopt:** Hum should treat componentization as successful only if it can compile a nontrivial package to a component; express file, clock, network, and tool access as explicit imports or resources; and pass a “no ambient authority” test suite that proves programs cannot reach the filesystem, environment, or network without declared capabilities. It should also hold an async gate separate from the rest: Hum should not commit its language-level async/effect story to WASIp3 until the relevant standards and toolchains are no longer explicitly transitional. Finally, because Rust’s `wasm32-wasip2` docs still say the target is not tested in CI, Hum should own its own cross-runtime CI for Wasmtime plus at least one second runtime or toolchain. cite

**What evidence would change the conclusion:** I would become more positive on a Wasm-first strategy if WASIp3’s async story stabilizes, the Rust/component target story moves beyond current transitional caveats, and Wasmtime’s incomplete component features shrink. I would become less positive if component debug-info and source-fidelity remain materially behind native targets, or if Hum’s contracts and blame model prove awkward to surface across component boundaries. cite

## The cost of keeping runtime checks in native code

**Finding:** The cost of retained runtime checks is **not one number**. The best evidence says overhead ranges from **very small** in some concretely optimized bounds-check cases to **very large** when checks accumulate in tight kernels or block optimization. The safest design conclusion for Hum is: keep checks by default, but make them easy to **prove away, hoist, merge, or replace with a smaller number of dominating checks**. **Confidence: High.** cite

**Research signals:** SPARK and Ada give the clearest long-lived systems-language evidence. GNATprove says it interprets annotations exactly as they are interpreted at runtime, including runtime checks, and verifies them statically. AdaCore’s SPARK guide says **95% to 98%** of runtime checks can typically be proved automatically, and that when checks are enabled they usually add **around 10%** running-time overhead, though heavily repeated hot code can incur much more. The same guide recommends proof as a better path than globally suppressing checks, because it lets programmers replace many checks with a smaller number of assumption checks while preserving safety. Ada’s assertion machinery is also explicit: `pragma Assert` can be enabled or ignored, and if ignored it has no runtime effect. AdaCore’s more recent overview says Ada runtime checks and contracts do have a code-size and performance footprint, but that SPARK’s value is precisely that proof can remove the need for paying that cost in production. cite

The modern microbenchmark evidence is more heterogeneous. The ASE paper on Rust performance reports average overheads of **1.96x** on its algorithm benchmarks and **1.35x** on its game benchmarks with checks enabled; disabling integer-overflow, bounds, and division-by-zero checks progressively lowers that, with out-of-bounds checks accounting for about **52.7%** of the total overhead in its algorithm set. But a NIST case study on C++ vector bounds checks found a best measured mean degradation of only **0.881%** for the checked form relative to the fastest unchecked construct it tested. These are not contradictions; they are a warning that check cost depends heavily on optimization context, data structure shape, and whether the check blocks vectorization, hoisting, or other profitable transformations. cite

The Rust paper’s overhead decomposition is worth seeing because it shows how much slowdown can survive even after some checks are removed. cite

For Hum’s “view invalidation” or dynamic alias/view checks, the evidence is thinner. I did **not** find a strong 2024-2026 primary benchmark literature specific to runtime view invalidation in a checked systems language with Hum-like views. The closest evidence is analogous. The Linear Dafny paper contrasts Rust’s `RefCell`, whose runtime borrow checks can panic and add time and space overhead, with linear/region-based techniques that avoid that runtime overhead entirely. That is relevant, but it is still an analogy rather than a direct measurement of Hum-style view invalidation. **Thin evidence: Medium confidence.** cite

AdaCore’s EMBENCH case study reinforces the “range, not scalar” point. It disabled Ada runtime checks to make a fair SPARK-vs-C comparison after proof, and still found many benchmarks clustered in single-digit overhead with one large outlier, which is broadly consistent with the idea that retained checks do not automatically dominate whole-program cost but can matter a great deal in particular kernels. cite

**Lessons for a new language:** Hum should design checks so they are **optimizer-friendly by construction**. Contract checks should be side-effect free, blame-producing failure paths should be cold, and `old()` snapshots should be explicit enough that the compiler can avoid gratuitous copying when a narrower snapshot suffices. Overflow checks should lower in a way that does not unnecessarily poison vectorization or LICM opportunities. View invalidation should be static wherever possible and dynamic only where the language deliberately allows patterns that static checking cannot express. SPARK’s larger lesson is important: the best long-term answer to retained checks is not a giant global “off” switch but a verification pipeline that lets you keep the few checks that matter and discharge the rest. **Confidence: High.** cite

**Concrete design gates or tests to adopt:** Hum should maintain a benchmark matrix with at least four classes: compute-heavy kernels, memory-heavy code, branchy business logic, and real toolchain workloads. Each class should be measured in four modes: interpreter, native with all checks, native with provably hoisted/merged checks, and native with checks suppressed only where a proof artifact exists. Hum should also collect per-check counters so it knows which checks dominate. Contract and overflow lowering should be rejected if they block vectorization or code motion in ways that exceed a predeclared budget on kernel benchmarks. For views, add a dedicated suite that stress-tests dynamic invalidation paths and measures the cost of generation counters or borrow-state metadata if Hum adopts them. Differential tests must confirm that every retained runtime check blames the same party and reports the same `old()`-aware message as the interpreter. cite

**What evidence would change the conclusion:** I would revise upward if Hum-specific measurements show retained checks stay within, say, a low-double-digit envelope on representative workloads and rare hotspots can be discharged by proof or hoisting. I would revise downward if Hum’s contracts or view checks systematically inhibit optimization the way bounds checks sometimes do in the Rust study, or if `old()` snapshotting causes persistent allocation/copy costs in macrobenchmarks. At present, the direct evidence on view invalidation remains thin, so Hum should assume risk and measure early. cite

## Lowering without losing source fidelity

**Finding:** If Hum wants trustworthy contract blame, debugger stepping, and Windows-first tooling, it should treat **source fidelity as a language-owned subsystem**, not a backend convenience. LLVM gives the strongest off-the-shelf story for rich debug info today; Cranelift gives a useful but currently thinner substrate centered on source locations, sequence points, debug tags, and stack-map/value-label metadata. **Confidence: High.** cite

**Research signals:** LLVM’s source-level debugging documentation is explicit that the frontend should capture how the important pieces of the source AST map onto LLVM code, and that optimizations should not be blocked by debug info. The Rust compiler dev guide shows the end-to-end shape of a mature frontend/backend debug stack: rustc inspects a language-owned IR, communicates relevant source, symbol, and type information to LLVM, LLVM lowers that to target-specific debug formats, and debuggers consume those formats. For Windows, the same Rust guide is candid that PDB/CodeView has a limited feature set rooted in C, that many details are undocumented or sparse, and that workarounds are common; LLVM’s own PDB docs explain why LLVM had to learn to generate PDBs directly in order to interoperate with the Windows tooling ecosystem. The practical conclusion is that Windows-native debugging is possible and real, but it is not something to hand-wave away. cite

Cranelift’s facilities are more focused. `FunctionBuilder::set_srcloc` lets a frontend assign a source location to new instructions. The IR records `srclocs` and says those source locations are preserved, not interpreted. It also has opaque `debug_tags` for sequence points and calls, and guarantees their structural behavior around inlining. Cranelift’s frontend API also exposes value labels and stack-map machinery useful for GC and moving-runtime cases. That is important and real. But the surrounding Wasmtime ecosystem still shows open edges: a 2024 Wasmtime issue asks for attaching source-language location information for clearer error reporting, and a 2026 Wasmtime issue reports breakpoints working for WASI HTTP/Spin Rust components while local variable locations are missing or “optimized out.” Those are materially relevant signals for Hum because they point to the difference between “I can map IPs back to lines” and “I have debugger-grade source fidelity end to end.” **Confidence: Medium-High.** cite

LLVM also offers a strong validation aid that Hum should copy into its process even if it uses Cranelift first: `llvm-debuginfo-analyzer` can print a **logical view** of debug information across ELF, Mach-O, WebAssembly, PDB, and COFF, compare logical views, and surface warnings about invalid lines, locations, or ranges. That kind of logical comparison is a better contract for a language team than “the binary contains some DWARF” because it tests whether the low-level records still describe the same source program. cite

**Lessons for a new language:** Hum should preserve a span-rich HIR or MIR that carries stable IDs for source constructs, including contracts, `old()` snapshots, and blame points. The interpreter and every compiled backend should consume this same metadata. In practice, that means contract failures should be modeled as first-class lowered operations with explicit source spans and explicit caller/callee/task-blame metadata, not as ad hoc inserted checks whose provenance the backend must reconstruct later. On Windows specifically, Hum should assume that polished PDB/CodeView output will take deliberate engineering, even if LLVM is doing most of the final encoding. The evidence strongly suggests that a backend-only mindset will not be enough. **Confidence: High.** cite

**Concrete design gates or tests to adopt:** Hum should add a source-fidelity test pack before it ships native binaries. The pack should verify breakpoint placement, single-stepping, variable inspection, stack traces, and contract-failure blame sites on both DWARF and PDB targets. It should also compare logical debug views across interpreter-generated traces, Wasm output, and at least one native path using `llvm-debuginfo-analyzer`. If Hum starts with Cranelift, it should require stable line mapping and blame mapping first, and postpone promises about rich local-variable inspection until they are actually demonstrated on Windows. cite

**What evidence would change the conclusion:** I would soften this view if Cranelift/Wasmtime demonstrates reliable local-variable and source-language attachment support across component and native contexts, especially on Windows. I would harden it if Hum discovers that `old()` state, contract lowering, or blame metadata cannot be traced cleanly through the generic debug formats without extra Hum-owned metadata and tools. cite

## Case studies on going from interpreter to native

**Finding:** The most relevant case studies do not say “ship the compiler as soon as it runs.” They say: **share semantics**, **fall back safely**, **test differentially**, and **set explicit go/no-go metrics**. The most common regret pattern is allowing users to depend on quirks of the slow path that the fast path either cannot or should not preserve. **Confidence: High.** cite

**Research signals:** OCaml remains the classical long-lived example. Current OCaml documentation says the same source should run identically when compiled through the bytecode and native compilers, which is a very strong promise. But a 1996 mailing-list reply about the then-optimizing native compiler shows exactly the kind of drift Hum should fear: code that “relies heavily on array bounds violation” was called out as ugly style that compromised compatibility with the optimizing compiler, and the reply explicitly mentions using an unsafe option to remove array bounds tests. That is a small but revealing historical regret signal: users will depend on whatever semantics your first implementation accidentally exposes. cite

Ruby’s YJIT shows the modern “fallback first” approach. The Ruby docs say YJIT lazily compiles hot instruction sequences, and if it hits an unexpected or unsupported operation it returns to the regular interpreter; behavior remains correct but slower. The side-exit architecture is not just a performance detail. It is a product strategy for controlling semantic risk. YJIT does not need to be perfect everywhere on day one because the interpreter remains available as the semantic backstop. cite

CPython’s JIT work makes the gating philosophy even more explicit. Python 3.13 added an experimental JIT that is disabled by default and uses a Tier 2 interpreter mainly for debugging the earlier optimization pipeline. Python 3.14 then shipped official macOS and Windows binaries including the experimental JIT, but still said it is not recommended for production use and that performance can range from **10% slower** to **20% faster** depending on workload. In 2026, PEP 836 goes further by proposing a time-bounded path under explicit performance, compatibility, tooling, distribution, security, and maintenance goals, including a concrete **20% geometric-mean performance target** by Python 3.17 or else reevaluation of whether the JIT should continue as an in-tree effort. That is unusually crisp evidence for how a mature project avoids “compiler optimism” and ties backend promotion to measurable outcomes. cite

For differential testing itself, the PLDI 2022 paper on interpreter-guided differential JIT compiler unit testing and its 2025 follow-up are directly relevant. The core idea is to use the interpreter as executable semantics, generate concrete tests by exploring interpreter paths, and then validate results through differential testing of interpreter and compiler/JIT. That research is not specific to Hum, but it is unusually well aligned with Hum’s interpreter-running-real-programs-today context. **Confidence: High.** cite

**Lessons for a new language:** Hum should internalize four lessons. First, the interpreter is not disposable scaffolding; it is the executable spec. Second, native backends should have explicit fallback or preview boundaries until they clear concrete gates. Third, the language spec should clearly separate intended semantics from accidental interpreter behavior. Fourth, backend promotion should be metric-driven, as in CPython’s current JIT roadmap, not ideology-driven. These are inferences from the case studies, but they are strongly supported. **Confidence: High.** cite

**Concrete design gates or tests to adopt:** Hum should build a permanent differential harness that runs the same corpus through the interpreter, the Wasm/component path, and the native path. Every mismatch should automatically reduce to a minimal reproducer and classify whether the divergence is in values, traps, blame assignment, source spans, or resource effects. Add interpreter-guided test generation specifically around contracts, moves, views, linear resources, and effect capabilities. Also adopt one CPython-like policy gate: a backend that does not hit a published semantic-compatibility target and a published performance target by a fixed release date stays experimental. cite

**What evidence would change the conclusion:** I would change this recommendation if Hum’s own semantics turn out to depend on behavior that is inherently hard to keep identical across interpreter and compiler tiers, such as intentionally observable evaluation-order details or timing-sensitive capability interactions. Otherwise, the existing evidence strongly favors interpreter-guided differentiation rather than backend duplication without a semantic oracle. cite

## A staged path that best fits Hum

**Finding:** The most evidence-backed path for Hum is: **interpreter as semantic ground truth**, **Wasm/components as the first serious compiled target**, **Cranelift native as the first direct native path for fast builds and selective deployment**, and **a willingness to add or retain an LLVM-based route later if Windows debug/PDB quality and release-mode performance demand it**. The LLVM sentence is an inference, not a direct roadmap from the sources, but it follows from the current debug-info and backend maturity evidence. **Confidence: Medium-High.** cite

**Research signals:** Cranelift is strong where Hum most needs early leverage: fast code generation, direct embeddability, and enough native execution to escape interpreter-only limits. Wasm/components are strong where Hum’s design story is most differentiated: explicit capabilities, deny-by-default authority, component imports/resources, and good sandboxable distribution. SPARK/Ada show that retained checks are commercially viable when combined with proof and selective suppression rather than all-or-nothing thinking. CPython, YJIT, and OCaml show that backend maturity should be staged and measured, not merely hoped for. cite

**Lessons for a new language:** Hum should optimize for **semantic leverage per engineering dollar**. That means the next year of backend work should prioritize whatever most improves correctness, testability, and user reach with the least irreversible commitment. Today that likely means: first, formalize backend-independent IR and blame metadata; second, make Wasm/components a crisp authority boundary; third, bring up Cranelift behind strong differential gates; and fourth, refuse to promise Windows-native toolchain polish until debugger and unwind gates are actually met. Saying “unknown” is appropriate here on one point: I found **no** primary 2024-2026 evidence showing a language with Hum’s exact mix of runtime contracts, checked ownership/views, and caller/task blame going through this path. So Hum should expect novel engineering rather than a full template. **Confidence: High on the process; Medium on exact implementation details.** cite

**Concrete design gates or tests to adopt:** Hum should publish backend promotion criteria now. A sensible first set would be: semantic mismatch rate effectively zero on the differential corpus; identical blame and `old()` behavior across backends; no ambient-capability escapes in the Wasm path; a debugger/source-fidelity scorecard that includes Windows; and explicit performance budgets for retained checks. For Cranelift on Windows, add a red-line gate: no graduation until unwind/exception behavior and debugger stepping are demonstrably correct. For components, add a red-line gate: no graduation until capability-denial and resource-handle tests pass in CI on pinned runtime versions. cite

**What evidence would change the conclusion:** The clearest game-changers would be a materially improved Windows Cranelift story, a more settled WASIp3 async story, and public Hum benchmark results showing retained checks stay affordable without weakening diagnostics. If those land, Hum could compress its path and market a much cleaner “interpreter to native” narrative. If they do not, the staged path above remains the safest reading of the evidence available today. cite
